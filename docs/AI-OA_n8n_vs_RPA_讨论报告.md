# AI-OA 工作流引擎讨论：n8n vs RPA

> 项目：AI-OA 智能化OA系统
> 日期：2026-04-05
> 参与：Master（项目经理）、Analyst（分析师）、Designer（设计师）、Coder（编码）

---

## 一、背景

财务审批模块（F2）需要工作流引擎支持：
- 多级审批流程
- 条件分支（金额阈值、部门类型）
- 人工/自动审批
- 定时任务

## 二、方案对比

### 2.1 n8n 方案

| 维度 | 评估 |
|------|------|
| **定位** | 开源工作流自动化平台 |
| **部署** | Docker/K8s 容器化 |
| **集成** | 200+ 官方集成（钉钉/企微/飞书等） |
| **API** | REST API + Webhook |
| **学习曲线** | 中等（可视化编辑器） |
| **费用** | 开源免费，自托管 |
| **定制能力** | 开源码，可深度定制 |

**优势**：
- 社区活跃，文档丰富
- 可视化流程设计，对业务人员友好
- 原生支持 Webhook Trigger
- 支持 Long-running workflows（人工审批节点）

**劣势**：
- Java SDK 支持有限
- 企业级特性（LDAP/SSO）需企业版
- 状态存储依赖自带 PostgreSQL

### 2.2 RPA 方案

| 维度 | 评估 |
|------|------|
| **定位** | 机器人流程自动化 |
| **部署** | 需部署 Agent 客户端 |
| **适用场景** | 桌面应用、遗留系统 |
| **API 集成** | 需通过屏幕录制模拟 |
| **稳定性** | 依赖 UI 层，版本更新易碎 |
| **费用** | 商业 license（UiPath/Blue Prism等） |

**优势**：
- 适合遗留系统/无 API 系统
- 模拟人工操作，100% 复现

**劣势**：
- 需要客户端部署
- API 集成能力弱
- 与 OA 系统架构不匹配
- 维护成本高

## 三、团队观点

### 3.1 Coder（编码）观点

**推荐 n8n**，理由：

1. **API 优先**：OA 系统基于 REST API，n8n Webhook Trigger 原生支持
2. **Long-running Workflows**：内置 Human Approval 节点，支持人工审批
3. **容器化友好**：Docker 一键部署，与 K8s 集成顺畅
4. **社区生态**：200+ Node 可直接复用，减少开发量

RPA 劣势明显：
- 需要在每台电脑上安装 Agent
- 无法 API 触发，适合桌面自动化而非企业流程
- 维护成本高，版本更新容易断裂

### 3.2 Designer（设计师）观点

**推荐 n8n**，理由：

1. **业务人员友好**：可视化编辑器，业务人员可自行调整流程
2. **版本管理**：流程变更可版本控制
3. **监控友好**：内置执行历史和监控

### 3.3 Analyst（分析师）观点

**推荐 n8n**，理由：

1. **成本**：开源免费 vs 商业 license
2. **集成**：原生支持飞书/钉钉/企微 API
3. **扩展性**：开源可深度定制

## 四、项目经理（Master）结论

**推荐方案：n8n**

| 决策 | 选择 |
|------|------|
| 流程引擎 | **n8n** |
| 部署方式 | Docker Compose / K8s |
| 集成方式 | REST API + Webhook |
| 审批节点 | n8n 内置 Human Approval Node |

**理由**：

1. ✅ n8n 是工作流引擎，与 OA 系统架构契合
2. ✅ RPA 是桌面自动化工具，适合遗留系统，不适合 API-first 的 OA
3. ✅ n8n 社区活跃，200+ 官方集成，落地成本低
4. ✅ 支持 Long-running workflows + Human approval
5. ✅ 容器化部署，与 K8s 集成简单

**备选方案**：

如未来需要 RPA 能力（对接遗留系统），可单独引入：
- 阿里云 RPA
- 实在智能 RPA
- 作为独立模块，不影响核心流程
